TL;DR

  • What it is: A full‑stack AI assistant with ingestion + retrieval‑augmented answering + observability + eval gates. [Evidence: README.md, server/index.ts, server/lib/retrieval.ts, server/lib/openai.ts, server/lib/observability/prometheus.ts, eval/golden/runner.ts]
  • Who it’s for: Teams who need grounded answers over connected sources (Google/Atlassian/Slack) and uploaded documents. [Evidence: README.md, server/lib/jobs/handlers/syncHandler.ts, server/lib/oauth.ts, shared/schema.ts]
  • Why it matters: It ships “guardrailable AI” (schema‑validated outputs, job retries + rate limits, traces/metrics, and regression eval gates). [Evidence: shared/schema.ts, server/lib/jobs/runner.ts, server/index.ts, server/routes_v2.ts, README.md]
Executive Snapshot
Users ask questions via chat; the server retrieves relevant chunks scoped to workspace/visibility, calls the model with a strict JSON schema, and returns answers with citations. [Evidence: server/routes_v2.ts, server/lib/retrieval.ts, server/lib/vectorstore.ts, shared/schema.ts]

Shipped workflow (end-to-end)

  1. Ingest data via upload (`/api/ingest`) or connector sync jobs (Google / Atlassian / Slack). [Evidence: README.md, server/routes_v2.ts, server/lib/jobs/handlers/syncHandler.ts]
  2. Normalize + store sources, immutable source versions, and chunks per workspace. [Evidence: shared/schema.ts]
  3. Index chunk embeddings in an in‑memory vector store with vectors persisted on chunks (`vectorRef`). [Evidence: server/lib/vectorstore.ts, shared/schema.ts]
  4. Ask in chat (`POST /api/chat`) and receive answers with citations/sources; trivial prompts use a fast-path. [Evidence: server/routes_v2.ts]
  5. Observe + evaluate with Prometheus metrics and eval runners (offline + golden) used in CI gating. [Evidence: server/lib/observability/prometheus.ts, eval/runner.ts, eval/golden/runner.ts, README.md]
RAG: top‑k + fallback retrieval
Answers: citations + schema
Ops: retries + rate limits
Quality: eval + CI gate

System design (high-level)

Key components: Web UI, Express API, Retrieval (vector + lexical fallback + neighbor expansion), Job Runner/Worker (sync + ingest), DB schema (sources/sourceVersions/chunks/jobs), Observability (Prometheus + traces/spans), OpenAI (GPT‑4o + embeddings). [Evidence: README.md, server/index.ts, server/routes_v2.ts, server/lib/retrieval.ts, server/lib/vectorstore.ts, server/lib/jobs/runner.ts, shared/schema.ts, server/lib/openai.ts, server/lib/observability/prometheus.ts]

Request flow (sequence)

The `/api/chat` route is a thin adapter over agent core and persists messages + citations. [Evidence: server/routes_v2.ts, shared/schema.ts]

Data lifecycle / pipeline

Embeddings use `text-embedding-3-small`, chat uses `gpt-4o`, and vectors can be persisted in `chunks.vectorRef`. [Evidence: server/lib/openai.ts, server/lib/vectorstore.ts, shared/schema.ts]

Reliability & safety (what exists)

  • Job retries + backoff + dead-letter for sync/ingest/eval/playbook jobs. [Evidence: server/lib/jobs/runner.ts, shared/schema.ts]
  • Concurrency + rate limiting per connector/account to protect external APIs. [Evidence: server/lib/jobs/runner.ts, shared/schema.ts]
  • Workspace/visibility boundaries in retrieval (private vs workspace; Slack private channel filtering). [Evidence: server/lib/retrieval.ts, shared/schema.ts]
  • Schema-validated outputs via Zod schemas (citations, actions, chat response shape). [Evidence: shared/schema.ts, server/lib/voice/websocket.ts]
  • Prod guard: server refuses to start with `EVAL_MODE=1` in production. [Evidence: server/index.ts]
  • Token handling: OAuth tokens are encrypted/decrypted via helper functions. [Evidence: server/lib/oauth.ts, server/lib/jobs/handlers/syncHandler.ts]

Reliability & safety (not found → recommended)

  • PII redaction policy: Not evidenced in repo → add structured redaction for logs/audit events (emails, phone numbers) and document retention policy. [Evidence: shared/schema.ts (audit_events), server/index.ts (API logging)]
  • Explicit model cost budget per request: Not evidenced in repo → enforce max tokens + refuse/ask to narrow scope on large context. [Evidence: server/lib/observability/prometheus.ts (token counters), shared/schema.ts (token fields)]
  • AuthZ for connectors: Some checks exist (workspaceId, visibility), but end-to-end connector permission mapping is not fully described → document and test scope enforcement. [Evidence: server/lib/retrieval.ts, shared/schema.ts]

2-minute demo script

  1. Start server: `npm run dev` (serves on port 5000). [Evidence: README.md, server/index.ts]
  2. Start worker: `npm run worker` (job runner polls, claims jobs). [Evidence: README.md, package.json, server/lib/jobs/runner.ts]
  3. Ingest a document (admin): open `/admin/ingest` or call `POST /api/ingest`. [Evidence: README.md, server/routes_v2.ts]
  4. Ask a question: open `/chat` and send; backend uses retrieval + schema output and persists citations. [Evidence: README.md, server/routes_v2.ts, server/lib/retrieval.ts, shared/schema.ts]
  5. Show observability: open `/admin/observability` and/or fetch `/api/metrics/prometheus`. [Evidence: README.md, docs/enterprise_readme.md, server/routes_v2.ts, server/lib/observability/prometheus.ts]
  6. Show quality gate: run `npm run eval` (golden runner writes reports) and explain CI thresholds. [Evidence: package.json, eval/golden/runner.ts, README.md]

Expected signals (examples): server logs request timing for `/api/*`; voice server logs “WebSocket server listening on /ws/voice”; job runner logs pending/claimed jobs. [Evidence: server/index.ts, server/lib/voice/websocket.ts, server/lib/jobs/runner.ts]
